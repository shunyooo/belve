# Design Review

変更が場当たり的でないか、設計として正しい方向に向かっているかをレビューする。

## いつ実行するか

- 同じ箇所を3回以上修正しているとき
- 修正が「前の修正の副作用を直す」パターンになっているとき
- 1つの変更が複数のファイル・レイヤーにまたがるとき

## レビュー手順

1. **変更の目的を1文で述べる**: 目的が1文で言えないなら、変更が大きすぎるか責務が混在している
2. **変更差分の取得**: `git diff` で全変更を確認する
3. **以下のチェック項目を検査する**

## チェック項目

### 責務の分離
- [ ] **各関数/ブロックの責務は1つか**: 1つの関数がセットアップ・接続・エラー処理を全部やっていないか
- [ ] **責務が混在していれば分離を提案**: 例）「セットアップと接続を分離すべき」
- [ ] **既存の責務分離を壊していないか**: 例）Service 層のロジックを View に引きずり出していないか

### 根本原因 vs 対症療法
- [ ] **問題の根本原因を特定しているか**: 「EOF が衝突する」→ なぜ EOF を使う必要がある？ → セットアップと接続が分離されていないから
- [ ] **修正は根本に対して行われているか**: ワークアラウンド（delimiter 変更、stty -echo 等）ではなく、構造を直しているか
- [ ] **同じ問題が別の形で再発しないか**: 今回の修正で類似の問題が他の箇所で起きないか確認

### 影響範囲の把握
- [ ] **変更が影響する全パスを列挙できるか**: 例）SSH 接続、DevContainer 接続、ローカル接続、reload、reconnect
- [ ] **変更していないパスが壊れていないか**: 他のプロジェクトタイプ、他のペイン数、他の操作で確認
- [ ] **共有リソースの競合がないか**: ファイル名、ControlMaster ソケット、tmux/persist セッション名の衝突

### 修正の積み重ね検出
- [ ] **この変更は「前の修正の修正」ではないか**: 修正→副作用→修正のループに入っていないか
- [ ] **3回以上の場当たり修正が発生していないか**: 発生していれば一度立ち止まり、設計を見直す
- [ ] **revert して別アプローチを取るべきか**: 現在の方向性が正しいか再評価する

### 設計の一貫性
- [ ] **既存の設計パターンに従っているか**: プロジェクト内の類似処理と同じパターンを使っているか
- [ ] **新しい概念や抽象化が必要最小限か**: 不要なヘルパー、中間ファイル、環境変数を導入していないか
- [ ] **将来の変更に対して脆くないか**: 例）ファイル名にインデックスを含めると、インデックス変更時に壊れる

## 出力フォーマット

```
## Design Review Result

### Status: PASS / FAIL / WARN

### Purpose
[この変更の目的を1文で]

### Design Issues
- **[MIXED_RESPONSIBILITY]** file.swift:N — セットアップと接続が同一関数に混在。分離すべき
- **[SYMPTOM_FIX]** file.swift:N — EOF 衝突を delimiter 変更で回避しているが、heredoc 自体が不要
- **[CASCADING_FIX]** 3回目の修正。前の修正（stty -echo）の副作用を直している。設計を見直すべき
- **[SHARED_RESOURCE]** /tmp/belve-ssh-0.sh を複数プロジェクトが共有。プロジェクト固有にするか、ファイル自体を不要にすべき
- **[FRAGILE]** pane index がファイル名に含まれるが、index は再起動で変わりうる

### Recommendation
[設計の方向性と具体的な次のステップ]
```
